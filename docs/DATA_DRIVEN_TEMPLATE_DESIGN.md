# Data Driven Template Design Reference

## Purpose

This document is the design reference for the first completed data-driven
template pass in `game_server`. It consolidates the policy now used by the
gameplay catalog, weapon templates, collider templates, and actor templates.

The actor-template policy follows
`docs/superpowers/specs/2026-06-11-actor-template-design.md`. This reference is
the stable place to check ownership boundaries before adding new gameplay data
fields.

## Design Goals

- Keep gameplay tuning in YAML instead of scattered hardcoded defaults.
- Keep `gameplay_catalog.yaml` as the composition root, not as a dumping ground
  for every gameplay number.
- Keep reusable facts in templates and scenario facts in catalog references.
- Preserve the existing kernel ABI unless a separate ABI design explicitly
  approves a larger migration.
- Keep template IDs stable and deterministic so packed catalog data, runtime
  references, tests, and catalog hashes stay comparable.
- Validate invalid references and malformed values at load time.
- Include all gameplay-affecting data in `compute_gameplay_catalog_hash`.

## Current Data Surface

The first version is organized as:

```text
game_server/gameplay_catalog.yaml
  -> weapon_template_dir: game_server/weapon_templates/*.yaml
  -> projectile_template_dir: game_server/projectile_templates/*.yaml
  -> actor_template_dir: game_server/actor_templates/*.yaml
  -> collider_template_file: game_server/collider_templates/default.yaml
  -> player.actor_template
  -> enemy.actor_template and enemy spawn policy
```

`gameplay_catalog.yaml` currently composes these data sources:

```yaml
catalog_version: 1
weapon_template_dir: weapon_templates
projectile_template_dir: projectile_templates
actor_template_dir: actor_templates
collider_template_file: collider_templates/default.yaml
player:
  actor_template: player
enemy:
  actor_template: enemy_grunt
  spawn_count: 1
  spawn_radius: 0.0
  spawn_seed: 4242
```

## Ownership Policy

Use this boundary before adding or moving fields:

| Data kind | Owner | Policy |
| --- | --- | --- |
| Catalog source paths | `gameplay_catalog.yaml` | Points to template files and directories. |
| Catalog version | `gameplay_catalog.yaml` | Version for data compatibility and hash input. |
| Weapon mechanics | `weapon_templates/*.yaml` | Reusable weapon behavior by stable weapon ID. |
| Projectile mechanics | `projectile_templates/*.yaml` | Movement, sync mode, lifetime, damage behavior, and collider template by projectile template ID/name. |
| Area/beam target rules | Weapon template mode blocks | Use `collision_mask`; do not add `collision_flags`. |
| Actor stats and loadout | `actor_templates/*.yaml` | Reusable actor definition by stable actor ID/name. |
| Player actor choice | `gameplay_catalog.yaml` | Reference an actor template by name or ID. |
| Enemy actor choice | `gameplay_catalog.yaml` | Reference an actor template by name or ID. |
| Enemy spawn count/radius/seed/position | `gameplay_catalog.yaml` | Scenario or encounter policy; not actor-template data. |
| Collider geometry and bindings | `collider_templates/default.yaml` | Kernel-packed collider templates and entity-type bindings. |

## Gameplay Catalog Policy

`gameplay_catalog.yaml` is the root composition file. It should answer:

- Which weapon template directory should be loaded?
- Which projectile template directory should be loaded?
- Which actor template directory should be loaded?
- Which collider template file should be loaded?
- Which actor template should the default player use?
- Which actor template and spawn policy should default enemies use?

It should not own:

- weapon damage, cooldown, projectile, beam, or area-effect mechanics
- actor HP, movement speed, hitbox, animation, AI, or weapon loadout
- collider shape geometry
- per-weapon friendly-fire flags
- kernel ABI layout changes

`weapon_template_dir`, `projectile_template_dir`, and `collider_template_file`
are required for catalog loading. `actor_template_dir` is supported by the first
actor-template pass. If actor templates are absent in compatibility paths,
defaults may still be generated by the loader, but new catalog data should
provide actor templates explicitly.

Player and enemy `actor_template` references may use a template name or numeric
ID. Names are friendlier in YAML; IDs are canonical for runtime storage and
deterministic tests. Duplicate actor template names or IDs must be rejected.

## Weapon Template Policy

Weapon templates own reusable weapon mechanics. A weapon template is identified
by a stable numeric `id` and a human-readable `name`.

Common weapon fields include:

```yaml
id: 0
name: Rifle
weapon_type: hitscan
magazine_size: 30
damage: 25
cooldown_ticks: 3
reload_ticks: 30
max_range: 100.0
```

Type-specific fields stay with their weapon mode:

- `hitscan`: range and direct-hit mechanics.
- `shotgun`: range, pellet count, and pellet spread.
- `projectile`: projectile movement, lifetime, sync mode, hit response,
  damage shape, explosion radius, gravity, and projectile collision mask.
- `area_effect`: area lifetime, radius, and area collision mask.
- `beam`: beam lifetime and beam collision mask.

Weapon templates should not own:

- actor loadouts
- spawn policy
- actor HP or movement speed
- scenario-specific enemy count
- player/enemy selection

### Collision Mask Policy

Use `collision_mask` as the single target-layer rule:

```text
collision_mask = target layers this weapon effect can hit
```

Do not introduce `collision_flags`. Friendly fire is represented by the mask:

```yaml
# Player weapon can damage enemies and players.
collision_mask: enemy | player

# Enemy weapon can damage players only.
collision_mask: player

# Projectile sync/simulation without damage hits.
collision_mask: none
```

Supported tokens are:

- `damageable`
- `enemy`
- `player`
- `projectile`
- `area_effect`
- `none`
- numeric `0`

The parser supports `|` expressions such as `enemy | player`. Unknown tokens
and empty tokens such as `enemy |` must be rejected.

Runtime policy:

- `collision_mask == 0` means the weapon effect can still exist, fire, and
  spawn/sync projectiles.
- Damage and hit queries must produce no damage targets for mask `0`.
- Projectile synchronization behavior must not depend on the mask being
  nonzero.

## Actor Template Policy

Use the term `actor template`.

Rationale:

- `actor` covers player, enemy, NPC, boss, turret, and other controllable or
  damageable gameplay entities.
- `entity template` is too easy to confuse with ECS internals.
- `character template` is too narrow for non-character gameplay actors.

The first actor-template implementation is game-server-local. It configures
runtime player and enemy combat state, but it is not packed into
`KernelGameplayCatalogDefinition` and does not change the kernel or Unity
managed ABI.

Actor templates own reusable actor facts:

```yaml
id: 2
name: enemy_grunt
entity_type: enemy
health:
  hp: 240
  max_hp: 240
movement:
  move_speed_meters_per_second: 2.5
hitbox:
  center: {x: 0.0, y: 0.8, z: 0.0}
  half_extents: {x: 0.4, y: 0.8, z: 0.4}
weapon_slots:
  - 2
active_weapon_slot: 0
animations:
  idle: 0
  chasing: 1
ai:
  profile: default
```

Actor templates should not own:

- enemy spawn count
- enemy spawn radius
- enemy spawn seed
- enemy spawn position
- encounter composition
- catalog source paths
- weapon mechanics
- collider binding policy beyond the actor's local hitbox/combat-state fields

### Actor Template Validation

Actor templates must be validated when loaded:

- `id` must be nonzero and unique.
- `name` must be non-empty and unique.
- `entity_type` is currently `player` or `enemy`.
- `health.hp` and `health.max_hp` must be positive, and `hp <= max_hp`.
- `movement.move_speed_meters_per_second` must be positive.
- `hitbox.center` and `hitbox.half_extents` are required.
- hitbox half extents must be positive on all axes.
- `weapon_slots` must contain 1 to 4 weapon IDs.
- every weapon slot must reference a loaded weapon template.
- `active_weapon_slot` defaults to `0` and must be in range.
- enemy AI derived from the actor template must reference a valid weapon and
  usable magazine/reload/fire timing values.

`weapon_slots` are actor loadout references, not embedded weapon definitions.
Ammo and runtime mechanics are derived from the referenced weapon templates.

## Collider Template Policy

Collider templates are already data-driven and kernel-packed. The current
collider data has two parts:

- `templates`: reusable collider geometry and layer/purpose metadata.
- `bindings`: entity-type to collider-template bindings with local positions.

Example shape:

```yaml
templates:
  - id: 2
    name: enemy_hit
    shape: aabb
    center: {x: 0.0, y: 0.0, z: 0.0}
    half_extents: {x: 0.4, y: 0.8, z: 0.4}
    radius: 0.0
    purpose: hit
    layer: enemy
bindings:
  - entity_type: enemy
    collider_template: enemy_hit
    local_position: {x: 0.0, y: 0.8, z: 0.0}
```

Current policy keeps collider binding by `entity_type`. Do not move collider
binding into actor templates unless a later design specifically needs multiple
actor archetypes of the same entity type to use different collider templates.

## Runtime Flow

Loading order:

1. Load the gameplay catalog root.
2. Load weapon templates from `weapon_template_dir`.
3. Load collider templates and bindings from `collider_template_file`.
4. Load projectile templates from `projectile_template_dir` and resolve weapon
   projectile references.
5. Load actor templates from `actor_template_dir`, using loaded weapon
   templates to validate actor loadouts.
6. Resolve `player.actor_template` and `enemy.actor_template`.
7. Compute the gameplay catalog hash from all gameplay-affecting data.
8. Validate the assembled `GameServerGameplayConfig`.

Player runtime:

1. Kernel emits `PlayerJoined`.
2. `GameServer::configure_player` resolves the configured player actor
   template.
3. `make_player_combat_state` builds combat state from actor stats and
   loadout.
4. Game server applies only the actor's configured weapon slots.

Enemy runtime:

1. `EnemyManager` resolves the configured enemy actor template.
2. Enemy placement uses `enemy.spawn_*` catalog settings.
3. The kernel entity is created with the actor template entity type and
   animation state.
4. `make_enemy_combat_state` builds combat state from actor stats and loadout.
5. Enemy AI uses template-derived weapon and movement settings.

## Kernel ABI Boundary

The first version intentionally keeps actor templates outside the kernel
gameplay catalog ABI. This means:

- weapons may still be packed into kernel weapon mechanics definitions
- projectile templates are loaded independently and packed for kernel use
- collider templates and bindings may still be packed for kernel use
- actor templates remain `game_server` configuration and runtime conversion
  data

Do not add `KernelActorTemplateDefinition`, extend
`KernelGameplayCatalogDefinition`, or update Unity managed ABI for actor
templates without a separate ABI design.

## Catalog Hash Policy

The gameplay catalog hash must change when gameplay-affecting template data
changes. It should cover:

- catalog version
- weapon template IDs, names, mechanics, mode-specific data, collision masks,
  projectile sync modes, and derived projectile data
- player and enemy actor template references
- enemy spawn position, count, radius, and seed
- actor template IDs, names, entity types, health, movement, hitboxes, weapon
  slots, active slots, animation states, and AI tunables
- collider template definitions
- collider bindings

Hashing should sort unordered template collections by stable IDs before
hashing, so filesystem enumeration order does not affect compatibility checks.

## Bundle Policy

Gameplay catalog bundles must include the catalog root and every referenced
template source needed to reconstruct the same config from memory:

- `gameplay_catalog.yaml`
- `weapon_templates/*.yaml`
- `actor_templates/*.yaml`
- `collider_templates/default.yaml`

Filesystem loading and bundle-memory loading should produce equivalent gameplay
configuration and catalog hash values.

## Extension Rules

Before adding a new field, answer these questions:

1. Is the field reusable across multiple actors, weapons, or colliders?
   Put it in the matching template.
2. Is the field scenario-specific or encounter-specific?
   Put it in `gameplay_catalog.yaml`.
3. Does the kernel need the field during simulation after loading?
   Pack it only through an existing kernel catalog structure unless a new ABI
   design is approved.
4. Does the field affect compatibility or gameplay outcome?
   Include it in the catalog hash.
5. Does the field reference another template?
   Allow friendly names when useful, but validate and store canonical IDs.

Deferred work:

- kernel-level actor template definitions
- Unity managed ABI exposure for actor templates
- actor-template-specific collider bindings
- team/faction/friendly-fire systems beyond collision masks
- broader encounter composition data

## Review Checklist

Use this checklist when reviewing template-related changes:

- `gameplay_catalog.yaml` remains a composition root.
- Weapon mechanics stay in weapon templates.
- Actor stats and loadouts stay in actor templates.
- Enemy spawn policy stays in the catalog, not in actor templates.
- Actor templates remain game-server-local unless an ABI design says otherwise.
- Collision targeting uses `collision_mask`, not `collision_flags`.
- `collision_mask: none` and `collision_mask: 0` still allow projectile
  spawn/sync while preventing damage targets.
- Template IDs and names are validated for uniqueness.
- Template references are validated at load time.
- Gameplay-affecting fields are included in catalog hash computation.
- Filesystem and bundle-memory catalog loading stay equivalent.
